# Contributing to Making Knowledge Plain

Thanks for contributing.

This project is not about making ideas shallow. It is about making them clearer, more enterable, and more accountable to the original text.

## What good contributions look like

A strong contribution usually does one or more of these:

- adds a high-quality rewrite example
- sharpens the project methodology
- improves evaluation or review standards
- adds benchmark references or prior art
- turns abstract principles into reusable templates

## Contribution types

### 1. Example analysis

Add a concrete case that includes:

1. original passage
2. the real question it is trying to answer
3. a faithful rewrite
4. optional explanatory layer
5. optional critique layer
6. a short review against the quality checklist

### 2. Method improvements

Improve rules, definitions, workflows, or review standards.

Examples:
- better problem taxonomy
- clearer distinction between rewrite vs explanation vs critique
- better domain-specific guardrails
- better review rubrics

### 3. Research additions

When adding external references, include:
- source link
- short summary
- what this project can borrow from it
- whether it changes any existing rule or standard

## Ground rules

### Preserve boundaries

Keep these layers separate:
- faithful rewrite
- explanatory help
- personal judgment

Do not mix them casually.

### Do not add unsupported meaning

If the original text does not support a claim, do not smuggle it into the rewrite.

### Do not flatten sharp ideas

A rewrite should be clearer, not softer.

### Do not replace one jargon with another

If you remove a term, do not hide it behind trendier language.

## Suggested workflow

1. Identify the source text and task type
2. Extract the core claim or problem
3. Check logic, conditions, and scope
4. Rewrite into plain but accurate language
5. Run the quality checklist
6. If needed, add explanation and critique as clearly separate sections

## File suggestions

- `docs/case-library/` for worked examples
- `docs/research-log-*.md` for ongoing external research
- `docs/templates/` for repeatable review or rewrite templates

## Pull request guidance

A good PR description should include:

- what problem this PR solves
- what files changed
- whether it affects principles, rules, evaluation, or examples
- what tradeoffs were made

## Standard

The standard is simple:

**Lower the barrier, not the meaning.**
