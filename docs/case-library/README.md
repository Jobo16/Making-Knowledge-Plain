# Case Library

This directory is for worked examples.

The goal is to make the project concrete.

Each case should show how a difficult passage gets handled, not just what the principles say in the abstract.

## Recommended case structure

Each case file should include:

1. Source metadata
   - title
   - author
   - date
   - link
   - domain
   - task type

2. Original passage

3. What problem the passage is actually addressing

4. Main claims

5. Faithful rewrite

6. Optional explanation layer

7. Optional critique layer

8. Review notes
   - fidelity
   - clarity
   - evidence visibility
   - concision
   - actionability

## Why this matters

A methodology-only project is easy to agree with but hard to use.

A case library shows:
- how the rules work in practice
- where the tradeoffs are
- what counts as a good rewrite
- how review decisions are made

## Suggested early categories

- philosophy / theory
- literary or art criticism
- AI / technology writing
- business / management language
- policy / public communication

## Standard

A case is useful only if another person can inspect it and understand:
- what the original said
- what changed
- why the rewrite is more enterable
- why it still remains faithful
